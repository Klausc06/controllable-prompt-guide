**Part A: Create init.ts barrel file**

Create `src/lib/prompt/init.ts` — a side-effect-only barrel file that imports all config modules in dependency order. This guarantees all registrations execute before any component renders or test runs (per research ARCHITECTURE.md Pattern 4).

```typescript
// src/lib/prompt/init.ts
// Import order matters: option sets must register before work types
// (work type validation checks optionSetId references against registry)

// 1. Option sets — register via options/index.ts module-level loop
import "./options";

// 2. Targets — register via targets/index.ts module-level loop
import "./targets";

// 3. Adapters — register via seedance.renderer.ts + generic-video.renderer.ts module-level calls
import "./renderers/seedance.renderer";
import "./renderers/generic-video.renderer";

// 4. Work types — register with question validation (checks optionSetId refs in registry)
import "./work-types/video-prompt.worktype";
```

**Import order rationale:**
1. Options register first (optionItemMap is built)
2. Targets register second (no dependency on options)
3. Adapters register third (they import target configs, triggering target registration via targets/index.ts side effects)
4. Work type registers last (its validation checks that optionSetIds exist in the registry — they were registered in step 1)

**Part B: Wire init.ts into layout.tsx**

Update `src/app/layout.tsx` to import init.ts as the first import (before any component that uses the registry):

```typescript
import "@/lib/prompt/init";  // Must be first — triggers all module registrations
import type { Metadata } from "next";
// ... rest of existing imports
```

Use a side-effect import (`import "@/lib/prompt/init"`) so tree-shaking does not remove it. The init.ts has no exports — it exists purely for import side effects.

**Verify:** After adding this import, `npm run build` must succeed. The import triggers all registrations before any component renders. The Next.js static export build verifies this works in production mode.

**Alternative consideration:** Instead of layout.tsx, could import init.ts in page.tsx. Layout.tsx is preferred because: (a) it runs before any page render including error states, (b) layout is the outermost server component, (c) it guarantees registration before PromptGuide mounts.
