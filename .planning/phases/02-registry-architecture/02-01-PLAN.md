**Part A: Wire options/index.ts**

Update `src/lib/prompt/options/index.ts` to self-register option sets into the registry (per D-01 and D-04):

1. Add import: `import { registerOptionSet } from "../registry";`

2. After the existing `optionSets` array and `optionSetById` computation, add a module-level loop that registers each option set. This runs at import time and detects duplicate option IDs:
   ```typescript
   for (const set of optionSets) {
     registerOptionSet(set);
   }
   ```
   This triggers `registerOptionSet()` which checks for duplicate set IDs AND duplicate option IDs across all sets (TEST-01, per D-04).

3. **CRITICAL: Keep all existing exports unchanged.** The `optionSets` array, `optionSetById` record, `getOptionSet()` function, and `getOptionById()` function must continue to work identically. The registry is an ADDITIONAL layer, not a replacement. This ensures backward compatibility:
   - Existing code calling `getOptionById(id)` continues to work (O(n) linear scan preserved)
   - The registry's `getOptionById` provides O(1) Map lookup for new code
   - `getOptionSet(id)` continues to work from `optionSetById` lookup

4. **Import order note:** The `registerOptionSet()` call happens at module evaluation time. ES modules guarantee that a module is only evaluated once. If tests use `vi.resetModules()`, re-importing options/index.ts would try to re-register the same sets. Current tests do NOT use `vi.resetModules()`, so this is not an issue. The registry test file (Task 1) tests registerOptionSet directly without importing options/index.ts.
